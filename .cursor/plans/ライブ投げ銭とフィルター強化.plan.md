# ライブ配信：投げ銭・フィルターの現状と強化方針（改訂）

## 方針変更：チップ（ギフト）金額はユーザー入力が主

固定プリセット（🎟100〜5,000）だけではなく、**視聴者がチケット数を自分で入力して決める** UI を主とする。

- **入力**: 数値フィールド（整数）＋送信前確認（確認モーダルまたは明確な「送る」ボタン）。
- **補助（任意）**: 「+100」「+500」「最大」などのショートカットはあってもよいが、**必須ではない**。あくまで入力が主。
- **サーバー**: 認証ユーザー必須。`giftAmount` は **最小値・最大値の範囲内**かつ**残高以下**であることを検証（プリセット一致は不要）。ホワイトリストは「許可された整数レンジ」に置き換える。

## 現状評価（参照）

### 投げ銭

- [`app/live/[id].tsx`](app/live/[id].tsx): プリセット4段のみ。サーバーは [`server/routes.ts`](server/routes.ts) の `POST /api/live-streams/:id/chat` で金額を検証・徴収していない。

### フィルター

- [`app/broadcast.tsx`](app/broadcast.tsx): ローカル `<video>` の CSS `filter` のみ。視聴者向けビットストリームへの反映は未検証／基本的にプレビュー用とみなす。

## 実装の段階

### フェーズ1（UX）

- ギフトモーダルを **カスタム金額入力**中心に再設計。残高表示（`/api/tickets/balance`）、不足時はチケット購入導線。
- **min / max** をプロダクト決定（例: 最小 10、最大 残高または上限 50,000 など）し UI と文言で明示。

### フェーズ2（サーバー）

- ギフト時: `getAuthUser`、残高チェック、原子デクリメント、`liveStreams` からクリエイター解決、`recordRevenue(..., "tip", ...)` 等へ接続。
- **金額検証**: クライアントの数値をそのまま信頼せず、整数・min/max・残高のみ許可。

### フェーズ3（フィルター）

- UI 改善と「プレビュー用」の説明。視聴者に届けるには Canvas `captureStream` または CDN 側など別タスク。
